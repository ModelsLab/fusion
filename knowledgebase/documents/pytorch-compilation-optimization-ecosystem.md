## 1. torch.compile

### 1.1 Architecture Overview

`torch.compile` is PyTorch's primary compilation API, introduced in PyTorch 2.0. It is a vertically integrated stack consisting of three major components:

```
Python Code
    |
    v
TorchDynamo (Graph Capture via Python bytecode rewriting)
    |
    v
AOTAutograd (Ahead-of-time backward graph capture)
    |
    v
TorchInductor (Code generation: Triton kernels, C++ kernels, CUDA templates)
    |
    v
Executable code (Triton -> PTX, C++ -> shared object)
```

**Basic usage:**

```python
import torch

@torch.compile
def fn(x, y):
    return torch.relu(x @ y + 1.0)

# Or compile an existing model
model = torch.compile(model)
```

### 1.2 TorchDynamo: Graph Capture via Python Bytecode

TorchDynamo is a Python-level JIT compiler that hooks into CPython's frame evaluation API (PEP 523) to intercept function execution. It rewrites Python bytecode dynamically to extract sequences of PyTorch operations into FX Graphs.

**How it works:**

1. **Bytecode Interception**: When a compiled function is called, CPython hands the frame to Dynamo via PEP 523's `_PyInterpreterState_SetEvalFrameFunc`.

2. **Symbolic Execution**: Dynamo reimplements CPython's stack machine through the `InstructionTranslatorBase` class (~200 methods handling Python bytecodes). Each bytecode instruction is symbolically executed:

```python
# Internal Dynamo handler example
def BUILD_LIST(self, inst):
    items = self.popn(inst.argval)
    self.push(ListVariable(items, mutation_type=ValueMutationNew()))
```

3. **Variable Tracking**: All Python values are wrapped in `VariableTracker` subclasses:
   - `TensorVariable` - PyTorch tensors (traced into FX graph)
   - `ConstantVariable` - Non-tensor, non-shape values
   - `ListVariable` - Python lists with mutation tracking
   - `UserDefinedObjectVariable` - Fallback for unrecognized types

4. **FX Graph Construction**: `OutputGraph` wraps `fx.Node` objects in `fx.Proxy` instances. Operations on proxies automatically record to the graph via `create_proxy` and `wrap_fx_proxy`.

5. **Guard Generation**: Guards are boolean assumptions that enable graph reuse without full retracing. They verify input properties at runtime:

```python
# Example guards generated by Dynamo:
__check_type_id(L['b'], 94334122025024)   # Type check
L['b'] == 'Hello'                          # Value check
L['x'].size()[0] == 32                     # Shape check
```

Guard **Sources** track how to reconstruct variables from original inputs:
- `LocalSource` - Direct function arguments
- `GetItemSource` - Indexed access like `list[0]`
- `AttributeSource` - Object attributes

**Compilation flow on subsequent calls:**
1. Guards checked against new inputs
2. If guards pass: execute pre-compiled code (cache hit)
3. If guards fail: retrace with new specialization (recompilation)

### 1.3 Symbolic Shapes

Three key properties define Dynamo's symbolic shape tracing:

1. **Static by Default**: Integers and shapes are initially treated as constants. Only after detecting shape changes across invocations does Dynamo trace them using `torch.SymInt`:

```python
# First call: shape [4, 3] -> constants baked in
# Second call: shape [8, 3] -> triggers recompilation with SymInt
```

2. **Zero/One Specialization**: Dimensions of 0 or 1 always generate specialized graphs because empty tensor behavior and contiguity depend on these edge cases.

3. **Duck Shaping**: When two dynamic integers match at trace time, Dynamo unifies them with a guard: `L['b'].size()[0] == L['a'].size()[0]`

**Enabling dynamic shapes:**

```python
# Explicit dynamic shapes
torch.compile(fn, dynamic=True)

# Per-dimension control via torch.export
from torch.export import Dim
batch = Dim("batch", min=1, max=128)
torch.export.export(model, (x,), dynamic_shapes={"x": {0: batch}})
```

### 1.4 Compilation Modes

```python
torch.compile(model, mode="default")                    # Balanced speed/compile time
torch.compile(model, mode="reduce-overhead")            # Uses CUDA Graphs, reduces Python overhead
torch.compile(model, mode="max-autotune")               # Maximum optimization, longer compile
torch.compile(model, mode="max-autotune-no-cudagraphs") # max-autotune without CUDA Graphs
```

| Mode | Description | Use Case |
|------|-------------|----------|
| `default` | Balances performance and compile time. Uses Triton for pointwise/reduction, cuBLAS for matmul. | General purpose, first attempt |
| `reduce-overhead` | Enables CUDA Graphs to eliminate Python/CPU overhead. May use more memory. | Small batch inference, latency-sensitive |
| `max-autotune` | Benchmarks multiple kernel implementations (Triton, CUTLASS, cuBLAS) per op. Includes CUDA Graphs. | Production deployment, willing to wait for compilation |
| `max-autotune-no-cudagraphs` | Same autotuning as max-autotune but without CUDA Graphs. | When CUDA Graphs cause issues (input mutations, dynamic shapes) |

**Additional compile options:**

```python
torch.compile(
    model,
    fullgraph=True,          # Error on graph breaks instead of falling back
    dynamic=True,            # Enable dynamic shapes
    backend="inductor",      # Default backend (can use custom)
    options={
        "triton.cudagraphs": True,           # Enable CUDA Graphs
        "max_autotune": True,                # Enable autotuning
        "epilogue_fusion": True,             # Fuse epilogues into matmul
        "coordinate_descent_tuning": True,   # Extra tuning passes
        "shape_padding": True,               # Pad tensors for better perf
    }
)
```

### 1.5 Graph Breaks

A **graph break** occurs when TorchDynamo encounters code it cannot capture into an FX graph. This splits execution into:
1. Compiled FX graph region
2. Eager Python execution (the "break")
3. Continuation function (compiled separately)

**Common causes of graph breaks:**

| Cause | Example | Solution |
|-------|---------|----------|
| Data-dependent control flow | `if tensor.item() > 0:` | Use `torch.cond` or restructure |
| `.item()` / `.tolist()` calls | `val = x.sum().item()` | Defer to outside compiled region |
| Unsupported Python operations | `print(tensor)` inside compiled fn | Remove or move outside |
| Third-party library calls | NumPy operations on tensors | Convert to PyTorch equivalents |
| Dynamic Python constructs | `exec()`, `eval()` | Avoid in compiled code |
| Tensor attribute access | `.data_ptr()` | Not needed in most cases |
| Non-standard data structures | Custom Python objects | Use supported types |

**Debugging graph breaks:**

```python
# Method 1: torch._dynamo.explain
explanation = torch._dynamo.explain(model)(example_input)
print(explanation)
# Shows: number of graph breaks, reasons, affected code lines

# Method 2: TORCH_LOGS
# TORCH_LOGS=graph_breaks python script.py

# Method 3: fullgraph=True (raises error on any graph break)
model = torch.compile(model, fullgraph=True)
```

**Preventing graph breaks:**

```python
# Mark functions to skip compilation (they run in eager)
@torch._dynamo.disable
def my_debug_function(x):
    print(f"Debug: {x.shape}")  # Would cause graph break if compiled
    return x

# Mark functions to always inline
@torch.compiler.allow_in_graph
def custom_op(x):
    return x * 2  # Guaranteed to be traced
```

### 1.6 TorchInductor: Code Generation

TorchInductor is PyTorch's default compiler backend that transforms FX graphs into optimized machine code.

#### 1.6.1 Architecture

```
FX Graph (from Dynamo/AOTAutograd)
    |
    v
Lowering Phase (FX ops -> Inductor IR)
    |
    v
Scheduling Phase (fusion decisions, tiling)
    |
    v
Code Generation (Triton / C++ / CUTLASS)
    |
    v
Autotuning (benchmark kernel variants)
    |
    v
Compiled Executable
```

**Inductor IR hierarchy:**

```
IRNode (abstract base)
├── Buffer (tensor storage)
│   ├── InputBuffer (graph inputs)
│   └── ComputedBuffer (operation results)
├── Operation (computable nodes)
│   ├── Pointwise (elementwise: add, relu, mul, exp)
│   ├── Reduction (sum, max, mean, amax)
│   └── MatMul (GEMM and batched variants)
├── Layout (tensor strides)
│   ├── FixedLayout (known at compile time)
│   └── FlexibleLayout (computed at runtime)
└── View (non-owning references)
    ├── PermuteView
    ├── ExpandView
    └── SqueezeView
```

#### 1.6.2 Fusion Rules

Fusion is TorchInductor's most important optimization. It turns multiple reads/writes into one read and one write -- critical for modern GPUs where memory bandwidth is the bottleneck.

**Pointwise-Pointwise Fusion:**

```python
# Before fusion: 3 kernels, 3 reads, 3 writes
y = x + 1          # Kernel 1: read x, write y
z = y * 2          # Kernel 2: read y, write z
out = torch.relu(z) # Kernel 3: read z, write out

# After fusion: 1 kernel, 1 read, 1 write
# Generated Triton:
@triton.jit
def fused_add_mul_relu(in_ptr0, out_ptr0, xnumel):
    xoffset = tl.program_id(0) * XBLOCK
    xindex = xoffset + tl.arange(0, XBLOCK)
    x0 = tl.load(in_ptr0 + xindex)
    x1 = x0 + 1
    x2 = x1 * 2
    x3 = tl.where(x2 > 0, x2, 0)  # relu
    tl.store(out_ptr0 + xindex, x3)
```

**Pointwise-Reduction Fusion:**

```python
# Before: separate pointwise and reduction
y = x * weight      # Pointwise
z = y.sum(dim=-1)   # Reduction

# After: fused into single kernel
# The multiply happens in registers before reduction
```

**Matmul Epilogue Fusion:**

When using `max-autotune` mode, Inductor can fuse operations into the GEMM epilogue:

```python
# Before: GEMM + separate bias add + ReLU
y = x @ weight      # GEMM kernel
y = y + bias         # Separate kernel
y = torch.relu(y)   # Separate kernel

# After: fused into GEMM epilogue
# cuBLAS/CUTLASS/Triton GEMM with epilogue:
# y = relu(x @ weight + bias) -- single kernel launch
```

**Reduction kernel strategies:**

TorchInductor selects between persistent and non-persistent reductions:

- **Persistent reduction** (reduction size <= 1024): Single Triton program loads full reduction axis. One global read + one global write. ~20% faster for small reductions.

- **Non-persistent reduction** (reduction size > 1024): Multi-pass with tiled accumulation. Uses `online_softmax_combine()` for streaming computation. Higher SM cycles due to multiple memory transactions.

```python
# Persistent reduction example (softmax with N=1024):
@triton.jit
def triton_per_fused_softmax(in_ptr0, out_ptr0, xnumel, rnumel):
    xoffset = tl.program_id(0) * XBLOCK
    # Load full tile: [XBLOCK, 1024]
    tmp0 = tl.load(in_ptr0 + (r0 + 1024 * x0), None)
    # Per-row max
    tmp_max = triton_helpers.max2(tmp0, 1)[:, None]
    # Normalize
    tmp_exp = libdevice.exp(tmp0 - tmp_max)
    tmp_sum = tl.sum(tmp_exp, 1)[:, None]
    out = tmp_exp / tmp_sum
    tl.store(out_ptr0 + (r0 + 1024 * x0), out, None)
```

**Kernel naming convention in traces:**

```
triton_poi_fused_cat_155      # poi = pointwise
triton_red_fused_sum_42       # red = reduction
triton_per_fused_softmax_0    # per = persistent reduction
```

#### 1.6.3 GEMM Backend Selection (max-autotune)

With `max-autotune`, Inductor benchmarks multiple backends for matrix multiply:

```
For a given mm/bmm/addmm:
├── cuBLAS (NVIDIA's optimized GEMM library)
├── Triton mm_template (block-tiled GEMM)
├── Triton persistent_mm_template (persistent reduction GEMM)
├── CUTLASS templates (NVIDIA's template library)
├── CK templates (AMD Composable Kernels)
└── Select fastest via benchmarking
```

Inductor peaks at ~706 TFLOPS on datacenter GPUs for large GEMM shapes with autotuning.

### 1.7 AOTAutograd

AOTAutograd (Ahead-of-Time Autograd / AOTDispatcher) captures the backward graph ahead of time, before any actual execution:

```python
# Conceptual flow:
# 1. Dynamo captures forward FX graph
# 2. AOTAutograd uses __torch_dispatch__ tracing to extract:
#    - Joint forward+backward graph
#    - Partitioned into separate forward and backward graphs
# 3. Both graphs passed to Inductor for optimization
```

**Key benefits:**
- Enables joint optimization of forward and backward
- Allows fusion across autograd boundaries
- Supports automatic activation checkpointing decisions
- Both forward and backward graphs compiled by Inductor

**Compiled Autograd (PyTorch 2.4+):** Extension that captures even larger backward graphs, including autograd engine dispatch logic:

```python
# Enable compiled autograd
with torch._dynamo.compiled_autograd.enable(torch.compile):
    loss.backward()
```

### 1.8 Caching

TorchInductor uses multi-level caching to avoid recompilation:

**Level 1 - Dynamo Cache:**
- Key: Python bytecode + guards
- Value: Resume functions + metadata
- Hit avoids re-running symbolic execution

**Level 2 - FX Graph Cache:**
- Key: FX graph structure + compilation flags
- Value: Compiled output code
- Persists across sessions via disk cache

**Level 3 - Inductor/Triton Code Cache:**
- Compiled `.cubin` files cached on disk
- Default location: `~/.triton/cache/` and `~/.cache/torch_inductor/`
- In production (e.g., vLLM): `~/.cache/vllm/torch_compile_cache/`
- Cache can be shared across machines with same environment

```python
# Pre-warm the cache (useful for deployment)
import torch._inductor.config
torch._inductor.config.fx_graph_cache = True       # Enable FX graph caching
torch._inductor.config.fx_graph_remote_cache = True # Remote/shared cache
```

## 2. CUDA Graphs in PyTorch

### 2.1 Fundamentals

CUDA Graphs (introduced in CUDA 10) capture a sequence of GPU operations as a single replayable unit, eliminating per-kernel CPU launch overhead:

```python
import torch

# Manual CUDA Graph usage
g = torch.cuda.CUDAGraph()

# Static inputs (addresses must not change)
static_input = torch.randn(1024, device="cuda")

# Warmup
s = torch.cuda.Stream()
with torch.cuda.stream(s):
    static_output = model(static_input)
torch.cuda.current_stream().wait_stream(s)

# Record
with torch.cuda.graph(g):
    static_output = model(static_input)

# Replay (fast - single CPU launch for all GPU ops)
static_input.copy_(new_data)  # Update input in-place
g.replay()                     # Replay recorded operations
# static_output now contains results
```

**Three fundamental constraints:**
1. **Asynchronous execution** - No CPU synchronization within the graph
2. **Static topology** - Fixed operation sequence at capture time
3. **Static parameters** - Kernel arguments and memory addresses must remain constant

### 2.2 CUDAGraphTree Architecture

CUDAGraph Trees (default in PyTorch) solve the problems of naive CUDA Graph usage through a tree-structured recording system:

**Key components:**
- `CUDAGraphTreeManager` - Per-device manager maintaining the tree structure
- `CUDAGraphNode` - Represents a single recording with parent/child relationships
- `CUDAWarmupNode` - Transient node for warmup phase (no graph recording)

**Tree structure (forest):**

```
# For conditional code:
@torch.compile(mode="reduce-overhead")
def foo(x):
    y = x * x
    if y.sum() > 0:
        z = y + 1       # Path A
    else:
        z = y - 1       # Path B
    return z * 2

# Results in tree:
Graph A (y = x*x)
├── Graph B (z = y+1) -> Graph D1 (out = z*2)
└── Graph C (z = y-1) -> Graph D2 (out = z*2)
```

**Execution flow (`CUDAGraphTreeManager._run()`):**

```python
def _run(self, new_inputs, function_id):
    # 1. Check for input mutations -> fallback to eager
    if self.non_cudagraph_managed_mutation_hint[...]:
        return self.ids_to_funcs[function_id].model(new_inputs)

    # 2. Warmup phase (if not warmed up)
    if function_id not in self.warmed_up_functions:
        if self.path_state == ExecutionState.EXECUTION:
            self.apply_checkpoint_execution_state_in_allocator()
        return self.run_eager(new_inputs, function_id)

    # 3. Try to find matching child node for replay (fast path)
    for child in child_nodes[function_id]:
        if child.check_invariants(new_inputs) == SUCCESS:
            return self.execute_node(child, new_inputs)

    # 4. No match -> restore checkpoint and record new branch
    if self.current_node is not None:
        self.apply_checkpoint_execution_state_in_allocator()
    return self.record_function(new_inputs, function_id)
```

### 2.3 Memory Management

**Checkpoint and Restore Mechanism:**

Each `CUDAGraphNode` saves allocator state after recording:

```python
# At end of recording
self.checkpointed_caching_state = torch._C._cuda_getCheckpointState(
    self.device, self.cuda_graphs_pool
)

# Before recording a new branch, restore parent's checkpoint:
def apply_checkpoint_execution_state_in_allocator(self):
    state = self.current_node.checkpointed_caching_state
    live_storages = list(self.current_node.path_live_weakrefs())
    torch._C._cuda_setCheckpointPoolState(device, state, [], live_storages)
```

**Liveness tracking via weak references:**

```python

    return [ref for ref in self.path_weakrefs if ref() is not None]

def data_ptrs_dead_since_invocation(self):
```

**Generation tracking (prevents stale access):**

```python

y1 = my_model(x)      # Output at address 0x1000

```

### 2.4 Dynamic Shape Handling

| Level | When | Mechanism | Config |
|-------|------|-----------|--------|
| Dynamo guards | First shape change | Full recompilation with new symbolic dimension | `automatic_dynamic_shapes` |
| FX-level | Unbacked shapes at compile | CUDA Graphs disabled entirely | `cudagraph_skip_dynamic_graphs` |
| int_key dispatch | Runtime varying shapes | New root node per unique int_key | `cudagraph_capture_sizes` |

```python
# For backed symbolic shapes (batch size, seq_len):
fn_cache: dict[tuple[int, ...], Callable] = {}

def deferred_cudagraphify(inputs):
    int_key = get_ints(inputs)  # Extract integers (batch_size, seq_len)

    if not is_cudagraph_capture_sizes(int_key):
        return model(inputs)     # Fallback if not in capture set

    fn = fn_cache.get(int_key)
    if fn is not None:
        return fn(inputs)        # Reuse for this int_key

    fn, out = cudagraphify(...)  # Record new graph
    fn_cache[int_key] = fn
    return out
```

### 2.6 Graph Partitioning

Not all operations are CUDA Graph compatible. Inductor partitions at:

- **Non-GPU operations** (CPU runs, device transfers)
- **`.cpu()` calls** (triggers `ir.DeviceCopy`)
- **`.item()` calls** (requires CPU-GPU synchronization -- disables CUDA Graphs entirely)
- **Conditional ops** (`torch.cond`, `torch.while_loop`)
- **Unbacked symbolic shapes** (data-dependent sizes like `x[x > 0]`)
- **Activation checkpointing** with random ops (`run_and_save_rng_state`)

```python
def fn(x):
    x = torch.relu(x)
    cpu_val = x.sum().cpu()       # Triggers partition (but doesn't disable CG)
    x = torch.softmax(x, dim=-1)
    return x, cpu_val
# Creates two cudagraphable partitions with eager execution between them
```

## 3. torch.export / ExecuTorch

### 3.1 torch.export

`torch.export` produces a fully traced, normalized graph for ahead-of-time compilation:

```python
import torch
from torch.export import export, Dim

class MyModel(torch.nn.Module):
    def forward(self, x, y):
        return x + y

model = MyModel()
example_inputs = (torch.randn(3, 4), torch.randn(3, 4))

# Static export
exported = export(model, example_inputs)

# Dynamic export with shape constraints
batch = Dim("batch", min=1, max=128)
exported = export(
    model,
    example_inputs,
    dynamic_shapes={"x": {0: batch}, "y": {0: batch}}
)

# Inspect the exported program
print(exported.graph_module.graph)
```

**Key characteristics of exported programs:**
- Only contains functional ATen operators (no Python control flow)
- All shapes are symbolic or static (no dynamic Python)
- Shape constraints are explicitly recorded
- Can be serialized/deserialized

**Export constraints:**
- No data-dependent control flow (`if tensor.item() > 0`)
- No Python side effects (print, file I/O)
- All operations must be traceable
- Dynamic dimensions must be declared upfront

### 3.2 AOTInductor

AOTInductor compiles exported models to shared libraries for deployment without Python:

```python
import torch
from torch.export import export

model = MyModel()
exported = export(model, example_inputs)

# Compile to shared library
so_path = torch._export.aot_compile(
    exported.module(),
    example_inputs,
    options={"aot_inductor.output_path": "/tmp/model.so"}
)

# Load and run without Python (in C++)
# torch::inductor::AOTIModelContainerRunnerCuda runner("/tmp/model.so");
# runner.run(inputs, outputs);
```

### 3.3 ExecuTorch

ExecuTorch targets edge deployment with minimal binary size and startup time:

```
torch.export -> ExecuTorch Intermediate Representation
    |
    v
Backend Delegation (XNNPACK, CoreML, QNN, Vulkan)
```

### 4.1 Modern Approach: torch.library (Recommended)

The modern way to add custom ops uses `torch.library` with `TORCH_LIBRARY` macros:

**Step 1: Define the operator schema**

```cpp
// myops.cpp
#include <torch/library.h>

TORCH_LIBRARY(myops, m) {
    m.def("mymuladd(Tensor a, Tensor b, float c) -> Tensor");
}
```

**Step 2: Register backend implementations**

```cpp
// myops_cuda.cu
#include <torch/library.h>
#include <ATen/ATen.h>

__global__ void muladd_kernel(
    const float* a, const float* b, float c, float* out, int n
) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        out[idx] = a[idx] * b[idx] + c;
    }
}

at::Tensor mymuladd_cuda(const at::Tensor& a, const at::Tensor& b, double c) {
    TORCH_CHECK(a.is_cuda());
    auto out = torch::empty_like(a);
    int n = a.numel();
    muladd_kernel<<<(n+255)/256, 256>>>(
        a.data_ptr<float>(), b.data_ptr<float>(), c, out.data_ptr<float>(), n
    );
    return out;
}

TORCH_LIBRARY_IMPL(myops, CUDA, m) {
    m.impl("mymuladd", mymuladd_cuda);
}
```

**Step 3: Register FakeTensor kernel (for torch.compile)**

```python
import torch
from torch.library import register_fake
```

**Step 4: Register autograd**

```python

    grad_b = grad_output * a
    return grad_a, grad_b, None  # None for float c

def mymuladd_setup_context(ctx, inputs, output):
```

### 5.1 CUDA Caching Allocator

### 5.2 PYTORCH_CUDA_ALLOC_CONF Options

```bash

# Or in Python
```

| Option | Default | Description |
|--------|---------|-------------|
| `backend` | `native` | Allocator implementation: `native` or `cudaMallocAsync` |
| `max_split_size_mb` | unlimited | Prevents splitting blocks larger than this size. Reduces fragmentation at cost of memory efficiency. Use as last resort. |
| `roundup_power2_divisions` | - | Rounds allocations to power-of-2 divisions for better reuse. Format: `[size_range]:[divisions]` |
| `garbage_collection_threshold` | 0.0 | Triggers GC when reserved memory exceeds `threshold * total_memory`. Range: 0.0-1.0. |
| `expandable_segments` | `False` | Uses CUDA low-level APIs (`cuMemCreate`/`cuMemAddressReserve`) to create segments that can grow. Significantly reduces fragmentation. **Recommended for most workloads.** |
| `pinned_use_cuda_host_register` | `False` | Uses `cudaHostRegister` for pinned memory |
| `pinned_num_register_threads` | 1 | Parallelizes page mapping for pinned memory |
| `per_process_memory_fraction` | 1.0 | Limits total allocatable GPU memory |

### 5.5 Activation Checkpointing

Trades computation for memory by recomputing activations during backward instead of storing them:

```python
from torch.utils.checkpoint import checkpoint

class TransformerBlock(nn.Module):
    def forward(self, x):
        # Without checkpointing: stores all intermediate activations
        x = self.attention(x)
        x = self.mlp(x)
        return x

class CheckpointedTransformerBlock(nn.Module):
    def forward(self, x):
        # With checkpointing: recomputes activations during backward
        x = checkpoint(self.attention, x, use_reentrant=False)
        x = checkpoint(self.mlp, x, use_reentrant=False)
        return x
```

**Selective Activation Checkpointing (SAC):**

```python
from torch.utils.checkpoint import checkpoint, create_selective_checkpoint_contexts

# Policy: recompute cheap ops (pointwise), save expensive ops (matmul)
def policy_fn(ctx, op, *args, **kwargs):
    if op in {torch.ops.aten.mm.default, torch.ops.aten.bmm.default}:
        return False  # Don't recompute matmuls
    return True       # Recompute everything else

# Apply selective checkpointing
context_fn = create_selective_checkpoint_contexts(policy_fn)
x = checkpoint(fn, x, use_reentrant=False, context_fn=context_fn)
```

**Reentrant vs Non-Reentrant:**

| Feature | Reentrant (legacy) | Non-Reentrant (recommended) |
|---------|-------------------|----------------------------|
| `use_reentrant` | `True` | `False` |
| Supports `torch.compile` | No | Yes |
| Supports custom autograd | Limited | Full |
| Thread safety | Issues possible | Safe |

## 6. Distributed Training / Inference in PyTorch

### 6.1 FSDP2 (Fully Sharded Data Parallelism)

FSDP2 uses DTensor-based per-parameter dim-0 sharding (replacing FSDP1's flat-parameter approach):

```python
import torch
from torch.distributed.fsdp import fully_shard, MixedPrecisionPolicy, CPUOffloadPolicy

# Basic FSDP2 usage
model = MyTransformer()

# Configure mixed precision
mp_policy = MixedPrecisionPolicy(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.float32,
)

# Apply FSDP2 per-module
for layer in model.layers:
    fully_shard(layer, mp_policy=mp_policy)
fully_shard(model, mp_policy=mp_policy)

# Training loop (same as non-distributed)
optimizer = torch.optim.AdamW(model.parameters())
for batch in dataloader:
    loss = model(batch)
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()
```

**FSDP2 vs FSDP1:**

| Feature | FSDP1 | FSDP2 |
|---------|-------|-------|
| Sharding | FlatParameter (concatenated) | DTensor (per-parameter) |
| API | `FullyShardedDataParallel` wrapper | `fully_shard()` in-place |
| Composability | Limited | Composes with TP, PP, torch.compile |
| Memory management | Uses record_stream | Deterministic release, lower memory |
| torch.compile support | Limited | Full |

**How FSDP2 works:**
1. Parameters converted to DTensor (sharded across ranks)
2. Pre-forward hook: all-gather parameters, convert DTensor -> plain Tensor
3. Forward pass executes with full parameters
4. Post-forward hook: free unsharded parameters, convert back to DTensor
5. Similar hooks for backward pass with reduce-scatter for gradients

### 6.2 Tensor Parallelism (DTensor)

DTensor is PyTorch's distributed tensor abstraction:

```python

    ColwiseParallel,

# Row-parallel: shard weight along input dimension

    "attention.wq": ColwiseParallel(),
    "attention.wk": ColwiseParallel(),
    "attention.wv": ColwiseParallel(),
    "attention.wo": RowwiseParallel(),

    "feed_forward.w2": RowwiseParallel(),

```

```python
quantize_(model, Int8DynamicActivationIntxWeightConfig(
    weight_dtype=torch.int4,
    weight_granularity=PerGroup(32),
```


### 9.7 AffineQuantizedTensor

The core tensor subclass for quantization in TorchAO:

```python
from torchao.quantization import AffineQuantizedTensor

# AffineQuantizedTensor stores:
# - int_data: quantized integer data
# - scale: per-group or per-channel scales
# - zero_point: quantization zero points
# - layout: how data is packed in memory

# Custom quantization via tensor subclass:
class MyQuantizedTensor(AffineQuantizedTensor):
    @classmethod
    def from_float(cls, float_tensor, group_size=32):
        # Custom quantization logic
        scale, zero_point = compute_qparams(float_tensor, group_size)
        int_data = quantize(float_tensor, scale, zero_point)
        return cls(int_data, scale, zero_point, ...)

    def dequantize(self):
        return (self.int_data - self.zero_point) * self.scale

    @classmethod
    def __torch_dispatch__(cls, func, types, args, kwargs):
        # Custom dispatch for quantized operations
        if func == torch.ops.aten.linear.default:
            return quantized_linear(args[0], args[1])
        ...
```

**Note:** AffineQuantizedTensor (v1) is being superseded by specialized tensor types (Float8Tensor, etc.) in v2 configurations.

## 10. FlexAttention

### 10.1 Overview

FlexAttention provides the flexibility to define arbitrary attention patterns in Python while compiling them to optimized Triton kernels with performance matching FlashAttention:

```
FlexAttention(Q, K, V) = softmax(mod(QK^T / sqrt(d_k))) V
```

where `mod` is composed of user-defined `score_mod` and `mask_mod` functions.

### 10.2 Core API

```python
from torch.nn.attention.flex_attention import flex_attention, create_block_mask

# Basic usage
output = flex_attention(
    query,           # (B, H, L, D)
    key,             # (B, H, S, D)
    value,           # (B, H, S, D)
    score_mod=None,  # Optional: modify attention scores
    block_mask=None, # Optional: block-level sparsity mask
    scale=None,      # Optional: custom scale factor
)
```

### 10.3 score_mod: Attention Score Modification

`score_mod` is a function that takes the attention score and position indices, and returns a modified score:

```python
def score_mod(score, b, h, q_idx, kv_idx):
    """
    Args:
        score: scalar attention score (q_i @ k_j / sqrt(d))
        b: batch index
        h: head index
        q_idx: query position
        kv_idx: key/value position
    Returns:
        Modified score (scalar)
    """
    return score  # Identity (full attention)
```

**Example: Causal attention**

```python
def causal(score, b, h, q_idx, kv_idx):
    return torch.where(q_idx >= kv_idx, score, float('-inf'))

output = flex_attention(query, key, value, score_mod=causal)
```

**Example: Relative position bias**

```python
def relative_position_bias(score, b, h, q_idx, kv_idx):
    bias = position_bias_table[q_idx - kv_idx + max_len]
    return score + bias

output = flex_attention(query, key, value, score_mod=relative_position_bias)
```

**Example: ALiBi (Attention with Linear Biases)**

```python
def alibi(score, b, h, q_idx, kv_idx):
    bias = -alibi_slopes[h] * abs(q_idx - kv_idx)
    return score + bias

output = flex_attention(query, key, value, score_mod=alibi)
```

**Example: Tanh Soft-capping (Gemma-style)**

```python
def tanh_softcap(score, b, h, q_idx, kv_idx):
    return 30.0 * torch.tanh(score / 30.0)

output = flex_attention(query, key, value, score_mod=tanh_softcap)
```

### 10.4 mask_mod and BlockMask

`mask_mod` defines which positions should attend to each other (boolean mask), and `create_block_mask` converts it to an efficient block-sparse representation:

```python
from torch.nn.attention.flex_attention import create_block_mask

def mask_mod(b, h, q_idx, kv_idx):
    """Returns True if position (q_idx, kv_idx) should attend."""
    return q_idx >= kv_idx  # Causal mask

block_mask = create_block_mask(
    mask_mod,
    B=batch_size,     # Can be None for batch-independent masks
    H=num_heads,      # Can be None for head-independent masks
    Q_LEN=seq_len,
    KV_LEN=seq_len,
    device="cuda",
    BLOCK_SIZE=128,   # Block granularity (default: 128)
)

output = flex_attention(query, key, value, block_mask=block_mask)
```

**How BlockMask works:**
- Divides the attention matrix into blocks (e.g., 128x128)
- For each block, determines if it's fully masked, fully unmasked, or partially masked
- Fully masked blocks are skipped entirely (no computation)
- Partially masked blocks apply `mask_mod` elementwise
- ~15% speedup for causal masks from skipping upper-triangle blocks

### 10.5 Use Cases with Code Examples

**Sliding Window Attention:**

```python
WINDOW_SIZE = 1024

def sliding_window(b, h, q_idx, kv_idx):
    return (q_idx - kv_idx).abs() <= WINDOW_SIZE

block_mask = create_block_mask(sliding_window, B=None, H=None, Q_LEN=L, KV_LEN=L)
output = flex_attention(q, k, v, block_mask=block_mask)
```

**Document Masking (packed sequences):**

```python
# document_ids: tensor mapping each token to its document
document_ids = torch.tensor([0,0,0,1,1,1,1,2,2], device="cuda")

def document_mask(b, h, q_idx, kv_idx):
    return document_ids[q_idx] == document_ids[kv_idx]

block_mask = create_block_mask(document_mask, B=1, H=1, Q_LEN=L, KV_LEN=L)
output = flex_attention(q, k, v, block_mask=block_mask)
```

**Prefix-LM (bidirectional prefix + causal suffix):**

```python
prefix_length = torch.tensor(256, device="cuda")

def prefix_lm(b, h, q_idx, kv_idx):
    # Bidirectional within prefix, causal after
    return (kv_idx <= prefix_length) | (q_idx >= kv_idx)

block_mask = create_block_mask(prefix_lm, B=1, H=1, Q_LEN=L, KV_LEN=L)
output = flex_attention(q, k, v, block_mask=block_mask)
```

**Causal + Sliding Window (combined):**

```python
def causal_sliding_window(b, h, q_idx, kv_idx):
    causal = q_idx >= kv_idx
    window = (q_idx - kv_idx) <= WINDOW_SIZE
    return causal & window

block_mask = create_block_mask(causal_sliding_window, B=None, H=None, Q_LEN=L, KV_LEN=L)
```

### 10.6 Composability

FlexAttention supports composition of `score_mod` and `mask_mod` functions:

```python
# Compose score_mods via nesting
def composed_score_mod(score, b, h, q_idx, kv_idx):
    score = alibi(score, b, h, q_idx, kv_idx)
    score = tanh_softcap(score, b, h, q_idx, kv_idx)
    return score

# Compose mask_mods via boolean operations
def composed_mask(b, h, q_idx, kv_idx):
    return causal_mask(b, h, q_idx, kv_idx) & document_mask(b, h, q_idx, kv_idx)
```

### 10.7 How FlexAttention Compiles to Efficient Kernels

1. **User defines** `score_mod` / `mask_mod` in Python
2. **torch.compile traces** these functions via Dynamo
3. **Triton kernel generated** with the score/mask logic inlined into the attention loop
4. **Block sparsity** exploited: BlockMask pre-computed to skip fully-masked blocks
5. **Autotuning** selects optimal Triton configurations (block sizes, num_warps)

```python
# FlexAttention MUST be used with torch.compile
flex_attention = torch.compile(flex_attention)

# Or compile the whole model
model = torch.compile(model)  # FlexAttention inside will be compiled
```

### 10.8 Performance vs FlashAttention

| Pattern | FlexAttention | FlashAttention-2 | Notes |
|---------|---------------|-------------------|-------|
| Full Attention | ~1.0x | 1.0x (baseline) | Equivalent |
| Causal | ~1.15x | 1.0x | Block skipping gives edge |
| Sliding Window | ~2-4x | N/A (requires custom kernel) | Sparsity exploitation |
| Document Masking | ~1.5-3x | N/A | Block sparsity varies by doc lengths |
| Custom Patterns | Supported | Not supported | FlexAttention's key advantage |

FlexAttention eliminates the "combinatorial explosion" of attention variants -- instead of writing a custom Triton/CUDA kernel for each combination (causal + ALiBi + sliding window + document masking), users compose simple Python functions.
